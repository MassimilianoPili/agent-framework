# ADR-001: Service Bus Topology — Unified Topic with SQL Filters

## Status

**Superseded** (original decision: one topic per worker type → revised to unified topic)

## Context

The agent framework dispatches tasks to specialized workers (BE, FE, AI-task,
contract, review). We need to decide the Azure Service Bus topology:

1. **Single topic with SQL filters**: One `agent-tasks` topic, workers use SQL
   subscription filters on `workerType` property
2. **One topic per worker type**: Separate topics (`agent-tasks-be`,
   `agent-tasks-fe`, etc.) with one subscription each
3. **Queue per worker**: Direct queues instead of topics

## Decision

We chose **Option 1: Single unified topic with SQL filters**.

## Rationale

- **Scalable without infra changes**: Adding a new worker type or profile requires
  only a new subscription with a SQL filter, not a new topic and infrastructure
  deployment.
- **Operational simplicity**: 3 topics total (`agent-tasks`, `agent-reviews`,
  `agent-results`) vs 5+ type-specific topics. Fewer resources to monitor and manage.
- **Consistent with runtime**: The `execution-plane/runtime/jobs/` job files and
  `topics.subscriptions.yml` already implement the unified topology — this decision
  aligns the specification with the operational ground truth.
- **SQL filter expressiveness**: Azure Service Bus SQL filters support compound
  expressions, enabling future routing by `workerProfile` without adding topics.
- **`agent-reviews` stays dedicated**: The review topic remains separate for QoS
  isolation (different lock duration, DLQ strategy, scaling characteristics).

## Trade-offs

- Workers must set `workerType` as a message property on every dispatch (already
  done by `AgentTaskProducer`).
- All BE/FE/AI_TASK/CONTRACT tasks share the same topic quota (1024 MB). Acceptable
  for current scale; can be raised without topology change if needed.
- A misconfigured SQL filter could cause missed messages — but this is caught at
  subscription creation time and is simpler to debug than routing bugs in producer code.

## Original Rationale (Option 2, now superseded)

The original decision chose separate topics for independent scaling, independent DLQ,
and simpler monitoring. In practice, the unified topology achieves the same goals:
Azure Container Apps Jobs scale independently per subscription (not per topic), and
each subscription has its own DLQ regardless of whether topics are shared.

## Consequences

- All task workers configure `task-topic: agent-tasks` in their `application.yml`
- `WorkerType.topicName()` returns `"agent-tasks"` for all types except REVIEW
- `AgentTaskProducer` sets `workerType` as a message property on every dispatch
- Bicep creates SQL filter rules on each subscription in `servicebus.bicep`
- DLQ replay is per-subscription (see `infra/azure/runbooks/dlq-replay.md`)
- Container Apps Jobs trigger on `agent-tasks` + their specific subscription

## Evolution: Per-Profile Subscriptions (February 2026)

The initial unified topology used **per-workerType** subscriptions (`be-worker`,
`fe-worker`, etc.) with SQL filters on the `workerType` message property. This was
sufficient when each workerType had a single implementation.

With the introduction of multi-stack profiles (be-java, be-go, be-rust, be-node for
BE; fe-react for FE), subscriptions have been refined to **per-profile** naming:

| Old (per-workerType) | New (per-profile) | SQL Filter |
|---|---|---|
| `be-worker` | `be-java-worker-sub`, `be-go-worker-sub`, `be-rust-worker-sub`, `be-node-worker-sub` | `workerProfile = 'be-java'` etc. |
| `fe-worker` | `fe-react-worker-sub` | `workerProfile = 'fe-react'` |
| `ai-task-worker` | `ai-task-worker-sub` | `workerType = 'AI_TASK'` (unchanged) |
| `contract-worker` | `contract-worker-sub` | `workerType = 'CONTRACT'` (unchanged) |
| `review-worker` | `review-worker-sub` | `workerType = 'REVIEW'` (unchanged) |

Benefits:
- Each profile scales independently (be-java can have 5 replicas while be-rust has 1)
- Granular DLQ per profile (be-go failures don't mix with be-java failures)
- `AgentTaskProducer` now also sets `workerProfile` as a message property
- Single-profile types (AI_TASK, CONTRACT, REVIEW) keep filtering on `workerType`

Source of truth for subscription naming: `agents/manifests/*.agent.yml` →
`config/worker-profiles.yml` (generated by `agent-compiler-maven-plugin`).
